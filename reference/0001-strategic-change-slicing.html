Strategic Change Slicing

A quick reference for turning ambiguous engineering work into small changes that still carry strategic intent.

A good slice is: one reason to change + one observable behavior or enabling constraint + one validation path + one rollback story.

The Four Questions

1. What outcome changes?

Name the user, operator, developer, or system behavior that should be different.

2. What is the smallest useful move?

Prefer a vertical slice when you need feedback. Prefer a horizontal slice when it unlocks several later slices.

3. How will we know it worked?

Attach tests, metrics, logs, screenshots, review checks, or manual QA steps to the slice.

4. How do we back out?

Keep configuration, data migrations, public contracts, and behavior changes separable when rollback risk is high.

Slice Types

  • Discovery slice: add logging, metrics, probes, or tests to reveal current behavior before changing it.
  • Contract slice: introduce or clarify an interface, schema, route, event, or boundary.
  • Vertical behavior slice: ship one user-visible path end to end.
  • Refactoring slice: change structure without intended behavior change, backed by tests.
  • Migration slice: move one caller, data segment, workflow, or traffic percentage to the new path.
  • Cleanup slice: remove dead code, obsolete flags, duplicate paths, or transitional scaffolding after evidence says it is safe.

Red Flags

A slice is too large when reviewers must understand several decisions at once, rollback would undo unrelated value, or validation depends on "trust me, I tested the whole thing manually."
  • Feature work and broad refactoring are mixed together.
  • The PR description says "phase 1" but does not explain the destination.
  • Tests are deferred to a later change even though behavior changes now.
  • The change adds an unused abstraction with no proving usage.
  • The old and new systems coexist, but no one knows when the old path can be deleted.
