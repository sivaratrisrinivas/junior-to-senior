Lesson 0001

Strategic programmers slice work before they code.

This lesson teaches one skill: taking an ambiguous, senior-shaped task and turning it into a small sequence of changes that can be reviewed, validated, and rolled back.

Why This Matters

AI can produce code quickly. That makes the bottleneck move upstream: deciding what should change, how much to change at once, how to validate it, and how to avoid trapping the team inside a risky half-migration.

Google's engineering practices argue for small, self-contained changes because they are easier to review, reason about, test, merge, and roll back. Fowler's writing on technical debt adds the strategic filter: cleanup is worth doing when it reduces the cost of future changes in code you will keep touching.

The core move: do not ask "Can I implement this?" Ask "What is the smallest change that creates evidence, preserves options, and improves the next change?"

The Slice Test

Before coding, a strategic slice should answer four questions.

  • Outcome: who or what benefits if this lands?
  • Boundary: what is included, and what is deliberately excluded?
  • Validation: what test, metric, QA step, or review evidence proves it worked?
  • Reversibility: how expensive is rollback if this is wrong?

Practice On Your Own Work

Bring your plan back to the agent and ask for a critique. The agent can help sharpen the slice sequence and spot hidden coupling.

Senior Signal

A junior often treats "small" as "less code." A strategic programmer treats "small" as "one coherent decision with a clear validation path."

Rule of thumb: if the reviewer must evaluate a product decision, an architecture decision, a data migration, and a refactor in the same PR, you probably skipped the slicing step.
